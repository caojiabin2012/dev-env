# 发版流程

本文说明从改版本号到 GitHub Actions 自动打包发布的完整步骤。

**仓库**：<https://github.com/caojiabin2012/dev-env>  
**默认分支**：`main`  
**触发条件**：推送形如 `v*` 的 tag（见 `.github/workflows/release.yml`）

---

## 一、改版本号

新版本号统一为 **SemVer**（如 `0.0.3`），以下文件必须保持一致：

| 文件 | 字段 |
|------|------|
| `package.json` | `"version"` |
| `src-tauri/Cargo.toml` | `[package].version` |
| `src-tauri/tauri.conf.json` | `"version"` |
| `src-tauri/Cargo.lock` | 根包 `dev-env` 的 `version`（改完 Cargo.toml 后执行 `cargo check` 会自动同步） |

> 应用内显示的版本来自 Rust `CARGO_PKG_VERSION`，与 `Cargo.toml` 一致。

---

## 二、写更新说明

CI 创建 Release 时会读取：

```
release-notes/v{版本号}.md
```

例如发 `v0.0.3`，需新建 `release-notes/v0.0.3.md`：

```markdown
## v0.0.3 更新说明

### 新功能
- ...

### 修复
- ...
```

建议同步更新根目录 `CHANGELOG.md`。

---

## 三、本地检查（可选但推荐）

```powershell
pnpm install
pnpm build
cargo check --manifest-path src-tauri/Cargo.toml
pnpm tauri build
```

确认能正常编译后再推送。

---

## 四、提交并推送到 GitHub

```powershell
# 查看改动
git status
git diff

# 暂存并提交
git add package.json src-tauri/Cargo.toml src-tauri/Cargo.lock src-tauri/tauri.conf.json release-notes/ CHANGELOG.md
git commit -m "release v0.0.3"

# 推送到 GitHub（origin 已指向 dev-env）

git push origin main


---

## 五、打 tag 触发打包

**tag 名必须为 `v` + 版本号**，且版本号与上文改的一致：

```powershell
# 创建 annotated tag（推荐）
git tag -a v0.0.3 -m "v0.0.3"

# 推送 tag 到 GitHub —— 这一步触发 Release Build 工作流
git push origin v0.0.3
```

也可一次推送多个 tag：

```powershell
git push origin v0.0.3 v0.0.4
```

---

## 六、查看工作流是否成功

### 方式 1：GitHub 网页

打开 Actions 页面：

<https://github.com/caojiabin2012/dev-env/actions/workflows/release.yml>

点击对应 run，确认三个构建任务均通过：

- `build-windows`
- `build-linux`
- `build-macos`
- `release`（汇总产物并创建 GitHub Release）

Release 页面：

<https://github.com/caojiabin2012/dev-env/releases>

应能看到安装包（`.msi` / `.exe` / `.deb` / `.AppImage` / `.dmg` / `.tar.gz`）以及 `latest.json`（应用内检查更新用）。

### 方式 2：GitHub CLI（`gh`）

先登录（仅首次）：

```powershell
gh auth login
```

查看最近的工作流运行：

```powershell
gh run list --repo caojiabin2012/dev-env --workflow release.yml --limit 5
```

实时跟踪最新一次运行（直到结束）：

```powershell
gh run watch --repo caojiabin2012/dev-env
```

查看某次运行的详情（将 `RUN_ID` 替换为上一条命令输出的 id）：

```powershell
gh run view RUN_ID --repo caojiabin2012/dev-env
```

若失败，查看失败步骤日志：

```powershell
gh run view RUN_ID --repo caojiabin2012/dev-env --log-failed
```

查看 Release 是否创建成功：

```powershell
gh release view v0.0.3 --repo caojiabin2012/dev-env
```

## 七、发版后验证

1. 在 Releases 页下载 Windows 安装包，安装并确认「关于 / 设置」中版本号正确。
2. 确认 `latest.json` 中 `version` 与 tag 一致，且各平台 `url` 可下载。
3. 在应用内点击「检查更新」，确认能识别新版本（需已发布且版本高于当前安装版）。

## 八、常见问题

### tag 已推送但 CI 没跑

- 确认 tag 格式为 `v0.0.3`（以 `v` 开头）。
- 在 Actions 页查看是否被禁用或排队。

### Release 创建失败：`release-notes/vX.Y.Z.md` 不存在

- 工作流 `body_path` 依赖该文件，发版前必须先创建。

### 改错了 tag 怎么办

未推送时可删除本地 tag：

```powershell
git tag -d v0.0.3
```

已推送且需作废（慎用，确认无人使用该 tag）：

```powershell
git push origin :refs/tags/v0.0.3
git tag -d v0.0.3
```

修正后重新打 tag 再推送。

如果需要继续使用同一版本号重新触发打包，也可以强制更新 tag：

```powershell
git tag -f v0.0.3
git push origin v0.0.3 --force
```

### 仅想本地打包、不触发 CI

```powershell
pnpm tauri build
```

产物目录：`src-tauri/target/release/bundle/`。
